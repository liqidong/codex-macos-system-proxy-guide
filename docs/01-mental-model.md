# 系统代理心智模型

这套方法的关键，是把“进入代理客户端”和“选择代理路线”分开看。

## 四个概念

### VPN

日常口语里的 VPN 经常泛指所有翻墙工具，但技术上 VPN 通常意味着系统级网络隧道。它可能会接管大量流量，甚至影响 DNS、局域网、开发服务器和系统服务。

### macOS system proxy

系统代理是 macOS 网络服务上的 HTTP / HTTPS / SOCKS 代理设置。支持系统代理的 App 会把请求交给这个本地代理入口。

它不是虚拟网卡，不会天然接管所有流量。影响范围通常比 TUN 小，但仍然会影响遵守系统代理的其他 App。

### 代理客户端

代理客户端负责监听本地端口，并根据规则把请求转发到不同代理组或直连。客户端名字不重要，重要的是它是否提供：

- 本地 HTTP / HTTPS 代理端口。
- 规则模式。
- 可持久化的规则覆写、增强、mixin、merge 或 profile enhancement 文件。
- 连接视图或日志，方便确认命中结果。

### TUN

TUN / 虚拟网卡 / 透明代理会让更多系统流量进入代理核心。它更强，也更重。它常和 DNS hijack、fake IP、增强 DNS 一起出现。

本仓库的默认判断是：如果只是让 Codex、Claude、OpenAI、Anthropic 相关流量稳定联网，优先使用系统代理，不把 TUN 作为第一选择。

## 推荐链路

```text
Codex / Claude
  -> macOS system proxy
  -> local HTTP proxy port
  -> proxy client rule engine
  -> real proxy group
```

这条链路里，需要你确认三件事：

- App 是否尊重 macOS 系统代理。
- macOS 系统代理是否指向正确的本地监听地址和端口。
- 代理客户端规则是否把 AI 服务域名送到正确代理组。

## 常见误区

“我开了代理客户端，所以 App 一定会走代理。”

不一定。代理客户端运行不等于系统代理开启，也不等于 App 会自动进入代理客户端。

“复制别人端口就可以。”

不可以。端口由你本机客户端决定，可能是 HTTP 端口，也可能是 SOCKS 端口，还可能每个配置不同。

“规则里写 Proxy 就行。”

不一定。代理组名必须是你配置里真实存在的组名。大小写、空格、emoji、地区名都可能不同。

“TUN 更强，所以应该打开。”

不一定。TUN 的影响面更大。如果系统代理路径能解决问题，先用轻量路径。
