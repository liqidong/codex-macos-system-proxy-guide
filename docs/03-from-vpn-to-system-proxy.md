# 从 VPN 经验迁移到 System Proxy

很多人第一次配置 AI App 联网，会把所有工具都叫“VPN”。这在日常交流里没问题，但做配置时容易混淆。

## 你真正需要判断的不是软件名

不要先问“我用哪个客户端”。先问：

```text
流量如何进入代理客户端？
代理客户端如何决定走哪个代理组？
规则配置能不能持久保存？
出了问题能不能观察和回滚？
```

本仓库选择的答案是：

```text
入口：macOS system proxy
路由：proxy client rule mode
持久化：user override / enhancement / mixin
观察：connection view / logs
回滚：disable system proxy + restore backup
```

## 为什么不直接全局代理

全局代理适合快速确认“节点能不能用”，但不适合作为教程最终状态。

全局代理的问题是：

- 它绕过了规则设计，不能证明 AI 域名规则正确。
- 它可能让本来应该直连的流量也走代理。
- 它不容易教别人定位问题。

规则模式才是可维护状态。

## 为什么不默认 TUN

TUN 适合处理不尊重系统代理的应用、复杂 DNS 场景或更强的全局接管需求。

但它也会引入额外变量：

- 虚拟网卡。
- DNS 劫持或 fake IP。
- 全系统流量接管。
- 局域网、开发服务器、同步服务异常。
- 更复杂的排障面。

如果 Codex 和 Claude 能通过 macOS system proxy 正常工作，就先不要把 TUN 加进来。

## 一套可复用流程

```text
1. 发现本机值
2. 确认规则模式
3. 关闭 TUN
4. 找到可持久化覆写文件
5. 备份
6. 写 AI 域名规则
7. 重载客户端配置
8. 开启 macOS system proxy
9. 用脚本检查
10. 打开 App
11. 看连接视图确认命中
```

这个流程的价值是可复制。换一个客户端、换一个代理组、换一个 App，也还是同一套判断。
